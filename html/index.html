Starburst - Helix

The hardware supplies the following memory hierarchy:

  • A local SRAM to every processor for single-cycle access to instructions and data.
  • A local memory that is write-only from the NoC and read/write from the local MicroBlaze. This memory is connected to the local PLB bus.
  • A global shared DDR SDRAM accessible for all processors, which can be cached by the MicroBlazes. This memory is connected to the NoC and is expensive in bandwidth, latency and power.

The software uses the memory hierarchy as follows:

  • All instructions are in DDR, which is cached; the .text section.
  • Half of the DDR is cached, which can be used for local data (the normal heap) and read-only data (.rodata section) and will be cached (without cache coherency).
  • The other half of the DDR is not cached, which can be used for global data (including the .bss section, .data and shared heap).

For applications, this means that an explicit separation must be made for these type of variables:

  • Function-local variables: allocate on the stack. Mind that the stack is small. The stack is malloc()-ed on the heap.
  • Application-local variables: you can use the stack, but this is limited in size, so use malloc() otherwise. The heap is cached and not cache coherent between cores.
  • Processor/kernel-local variables: the processor heap is shared among all processes, so this can be used for inter-process communication.
  • Global variables: use the shared heap (smalloc()) for that. Data will be put in uncached DDR, accessible by all cores, but very slow.

Misc notes:

  • When passing pointers between processes, they must be in uncached memory or the data cache must explicitly be flushed/invalidated.
  • Read-only data is punt in the cached .rodata section, read/write data in uncached .data. If a static variable is used as read-only, define it as 'const'.
  • There is no memory protection; writing to wrong addresses (including NULL) can result in really strange behavior (even after reset).
  • Stack overflows will not be detected sometimes, so keep you stack usage low and inspect it when necessary.
  • C++'s operator new() and operator new[]() are redirected to smalloc(), and operator delete() and operator delete[]() to sfree().
  • C++'s exceptions, virtual functions, etc. is kind of supported but not really tested.

Compilation hints:

  • The entry of your program is main(), as usual. This process is automatically started at core 0 when the system boots.
  • Compile everything statically.
  • Link against libhelix.a (which is done automatically when using starburst/bin/mb-g++).

Running daemons at the time main() starts:

  • DefaultTaskDaemon() (always process id 0)
  • MallocDaemon()
  • StartDumpStatsDaemon()
  • StartTimerDaemon() when you use a timer or sleep()
  • mainProc() which calls main() eventually
